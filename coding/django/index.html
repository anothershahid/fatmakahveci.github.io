manage.py

  • It points to settings.py.

The Django MVT (model view template) pattern

user <=> django <=> url <=> view <=> model
                                 <=> template

The order, in which you create Templates, Views, and URLs, doesn’t matter since all works together closely.

1. Model manages data and core business logic.

2. View describes which data is sent to the user.

3. Template presents the data as HTML.

4. URL dispatcher is equivalent to the controller in the MVC architecture.

  • Django creates a primary key automatically for each model, but you can also override this by specifying primary_key=True in one of your model fields.

Migrations

$ python manage.py makemigrations <app_name> applies changes in your model.

$ python manage.py migrate applies changes to the database. It reads 001_initial.py, and creates and run SQL commands.

$ python manage.py shell is different than the UNIX shell.

$ python manage.py showmigrations displays the details.

Migrations is a logical description of what the tables should look like. They are guided by the installed applications listed in settings.py.

migrate command reads all the migrations folders in the application folders and creates/evolves the tables in the database (i.e. db.sqlite3).

  • Note: If you get kind of messed up, you can remove <file>.sqlite and run the migrations again.

Migration class typically contains two important sequences:

  • Dependencies refer to the previous migration.
  • Operations refer to the actions performed in the given migration.

2. View

  • It determines what content is displayed on a given page.

  • It interacts with both the model and template layers.

  • There are three ways to write a view in Django:

    • 1. Function-based view takes an HttpRequest object and returns an HttpResponse object. Both are defined in django.http module.
      • The view function receives the request data in an instance of class HttpRequest.
        • All the data coming in.
        • Data you see in your browser debugger about what data was sent to the server, that all ends up somewhere in this object.
      • An instance of HttpResponse class
        • It contains the actual contents of default content_type as “text/HTML” and the status code.
        • Something that we create inside of our view. It might be simply text (HTML). (OR)
        • You send it all back. It might be a redirect object, go to a different page.
    • 2. Class-based view allows for much greater code reusability and keeps things DRY (Don’t Repeat Yourself). It inherits from the View class. You always add as_view() at the end of the view name, when using class-based views.
    • 3. Generic class-based view handles common use cases, such as creating a new object, forms, list views, pagination, etc. They are built-in.

  • Django request/response cycle: URL -> View -> Model -> Template

# <app_name>/views.py
from django.http import HttpResponse

  return HttpResponse("Hello World!")

4. URLs

A URL pattern path(<string>, <view>, <name>) allows you to map URLs to views.

User <-> Django <-> urls.py (project) <-> urls.py (app) <-> views.py

If the user requests the homepage, represented by the empty string "", then use the view called homePageView.

URLConf maps request to the appropriate view function which returns the correct data. It is to design URLs for an app. It maps URL path expressions to python functions.

Reading the URL

  • <website_url>/views/<app>
  • <website_url>/views/<app>?<key>=<value>(parameters GET)
  • <website_url>/views/<app>/<url_path_parameter>

Three patterns for views (in urls.py)

  1. Requests are routed to a predefined class from Django itself.
  2. Requests are routed to a function in views.py that takes the HTTP request as a parameter and returns a response.
  3. Requests are routed to a class in views.py that has get() and post() methods that take the HTTP request as a parameter and return a response.
  • URL Mapping / Reversing allows us to generate links from one page to another.
  • reverse() builds URLs by their name and pass optional parameters.

QuerySet API

QuerySet API interacts with the database.

A QuerySet represents a collection of objects from the database.

  • <Model>.objects.all() := SELECT * FROM <Model>;
  • <Model>.objects.filter(<criteria>) := SELECT * FROM <Model> WHERE <criteria>;
  • <Model>.objects.filter(<field>__startswith='c') := SELECT * FROM <Model> WHERE <field> LIKE "c%";
  • <Model>.objects.exclude(start_date__gt=datetime.date(2022, 2, 2), header='Header') := SELECT * FROM <Model> WHERE NOT (start_date > '2022-2-2' AND header = 'Header')
  • <Model>.objects.exclude(start_date__gt=datetime.date(2022, 2, 2)).exclude(header='Header') := SELECT ... WHERE NOT start_date > '2022-2-2' AND NOT header = 'Header'

  • Form arguments:

    • required := True (by default)
    • label := specify a label for the field
    • initial declares the initial value of form fields at runtime.
    • help_text is a descriptive text for the field.
    • widget defines how the content will be rendered as HTML.
from django import forms

COUNTRY = [ ('italy', 'Italy'), ('turkey', 'Turkey'), ]

class DemoForm(forms.Form):
  age = forms.IntegerField(required=False)
  email = forms.EmailField(initial="foo@bar.com")
  country = forms.ChoiceField(choices=COUNTRY)
  publication_date = forms.DateField(help_text="A valid date is required.", widget=NumberInput(attrs={'type':'date'}))
  comment = forms.CharField(widget=forms.Textarea(attrs={'rows':5}))

Sending emails

  • A local SMTP server or an external SMTP server with the following is required.
    • EMAIL_HOST := default: localhost
    • EMAIL_PORT := default: 25
    • EMAIL_HOST_USER := server username
    • EMAIL_HOST_PASSWORD := server password
    • EMAIL_USE_TLS := whether to use Transport Layer Security (TLS)
    • EMAIL_USE_SSL := whether to use a secure TLS connection
